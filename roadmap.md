# Roadmap

## Vision

构建一个面向人工调度的 AI 工程交互终端。项目本身不自动决定哪些 issue 开始、哪些 issue 暂停，而是在人工选择任务后，辅助完成需求理解、代码修改、验证、code review、提交 pull request，并在主分支更新后维护已提交 PR 的 rebase 状态。

## Product Boundary

- 当前项目提供交互终端能力，不提供全自动 issue 调度系统。
- 是否开始处理某个 issue 必须由人工决定。
- 终端可以展示 issue 信息、辅助分析任务和执行工程动作，但不能绕过人工授权自动启动任务。
- 自动化能力应聚焦在人工确认后的执行链路，包括修改代码、验证、review、提交 PR 和维护 PR 分支。

## Target Workflow

1. 人工在交互终端中选择要处理的 issue 或输入明确任务。
2. 终端读取任务上下文，辅助识别问题类型、影响范围和验收标准。
3. 基于仓库现有架构与规范分析代码，定位需要修改的模块和测试范围。
4. 修改代码并补充必要测试，避免破坏既有架构边界。
5. 执行本地验证，至少覆盖 lint、test 和与变更相关的专项检查。
6. 对改动进行 code review，识别潜在 bug、回归风险、缺失测试和文档同步问题。
7. 在人工授权后创建分支、提交变更并打开 PR，PR 描述包含问题背景、实现摘要、验证结果和风险说明。
8. 监听主分支更新，在 PR 分支落后时自动 rebase，并重新执行必要验证。
9. 在 rebase 冲突或验证失败时暂停自动推进，输出明确的失败原因和人工处理建议。

## Milestones

### M1: Interactive Task Intake

- 提供交互终端入口。
- 支持人工选择 issue 或输入明确任务。
- 提取需求、复现信息、约束、影响模块和完成标准。
- 在需求不明确时向人工追问，而不是自动启动任务。

### M2: Code Change Agent

- 根据 issue 自动搜索现有实现。
- 生成最小可行修改方案。
- 修改代码、测试和必要文档。
- 遵守仓库的架构、命名、注释和测试规范。

### M3: Verification And Review

- 自动运行仓库标准验证命令。
- 汇总失败日志并尝试修复明确问题。
- 对最终 diff 做 code review。
- 阻止带有高风险问题的 PR 自动提交。

### M4: Pull Request Automation

- 创建任务分支并提交变更。
- 自动生成 PR 标题和正文。
- 在 PR 中记录关联 issue、实现摘要、验证结果和残余风险。
- 提交和创建 PR 前需要人工授权。

### M5: Main Branch Rebase Automation

- 监听主分支更新事件。
- 检测已打开 PR 是否落后于主分支。
- 自动 rebase PR 分支并重新运行验证。
- 处理成功后更新 PR 状态，遇到冲突或测试失败时请求人工介入。

## Acceptance Checklist

- [ ] 能够通过交互终端接收人工选择的 issue 或明确任务。
- [ ] 不会自主决定开始处理哪些 issue。
- [ ] 能够从人工选定的 issue 中提取明确的工程任务。
- [ ] 能够在不违反现有架构边界的前提下修改代码。
- [ ] 能够自动补充或更新相关测试。
- [ ] 能够在提交 PR 前完成自检和 code review。
- [ ] 能够创建包含完整上下文的 PR。
- [ ] 能够在主分支更新后自动 rebase 已提交 PR。
- [ ] 能够在冲突、验证失败或需求不明确时安全停止并报告原因。

## Open Questions

- 是否只支持 GitHub，还是需要预留 GitLab 等平台适配层。
- 人工授权的粒度如何定义，例如开始任务、提交 commit、创建 PR、执行 rebase 是否分别确认。
- PR 自动 rebase 后需要运行哪些最小验证集合。
- 何种风险级别必须转人工 review。
