## 待澄清问题
1. 路线图的具体呈现形式：是甘特图、时间轴、网络图，还是列表/看板？（来源：2026-06-14 19:07）
   - **推荐：时间轴 + 列表双视图，先不做甘特图/网络图。** 理由：用户原话同时提到「一个顺序，一个路线图」，时间轴能表达顺序与依赖，列表适合快速筛选与定位；甘特图强依赖起止日期字段，网络图实现与交互成本高，二者都可以后续再补。
2. PRD 依赖关系的数据来源：是否从 PRD frontmatter 中解析，还是另外维护依赖文件？（来源：2026-06-14 19:07）
   - **推荐：从 PRD frontmatter 解析，不另建维护文件。** 理由：PRD 本身已经是唯一事实来源，新增 `depends_on` / `related` 等字段即可让依赖随 PRD 移动、归档自然同步，避免多文件维护带来的不一致。
3. 「全部 / 未完成」开关的默认状态，以及归档 PRD 在视图中的视觉层级如何区分？（来源：2026-06-14 19:07）
   - **推荐：默认只看未完成 PRD，归档项可展开/置灰显示。** 理由：与原话「最主要的是看没有做完的PRD」一致，减少视觉噪音；归档 PRD 通过置灰、折叠或完成图标区分，既满足「也可以加在里面」又不喧宾夺主。
4. 是否需要支持拖拽调整顺序/依赖，还是只读展示？（来源：2026-06-14 19:07）
   - **推荐：第一阶段只读展示，后续再评估拖拽编辑。** 理由：先验证「看状态、看依赖」的核心价值，避免一开始就引入拖拽排序、依赖修改的状态同步和并发写入复杂度；如果看了一段时间确实需要调整，再升级。
5. 路线图上的「开始 PRD」操作是触发什么：创建分支/worktree，还是仅仅标记状态为 in_progress？（来源：2026-06-14 19:41；**用户确认：2026-06-14 19:45**）
   - **已确认：点击「开始 PRD」要真正开始工作 — 修改 GitHub label（level）并同时在本地创建 worktree。** 说明：这是用户的明确要求，不是仅做状态标记；实现时需要集成 GitHub API 变更 label 以及调用本地 worktree 创建逻辑，同时要注意与常驻 iar daemon 的并发竞争问题。
6. 「需要 review / 去 PR / 合并 PR」这些状态由谁驱动？（来源：2026-06-14 19:41；**用户确认：2026-06-14 19:46、2026-06-14 19:48**）
   - **已确认：由 Agent 驱动实现、内部 review 和创建 PR；人主要负责确认开始和审阅 PR。** 说明：项目现有流程中 Agent 已经能完成实现到 Draft PR 的流转，前端路线图应展示这些状态并高亮待人介入的节点，而不是重新实现状态检测逻辑。
7. 「完成之后进行下一个」是自动开始下一个 PRD，还是仅高亮下一个可开始的项？（来源：2026-06-14 19:41；**用户确认：2026-06-14 19:53**）
   - **已确认：提供「全局开始」和「单个开始」两种模式。**
     - **全局开始**：用户点击后设置并行数量；无依赖的 PRD 可并行执行；有依赖的 PRD 需等待上游被审阅并合并到主分支后才能开始。
     - **单个开始**：用户点击单个 PRD 开始执行。
   - 说明：这需要路线图后台具备依赖拓扑调度能力，并与 iar daemon 的并发执行、worktree 资源管理协调。
8. 这个工作流与现有 iar / PR 自动化流程的关系是什么？是替代、封装还是只读展示？（来源：2026-06-14 19:41；**用户确认：2026-06-14 19:54**）
   - **已确认：是对现有 iar / PR 自动化流程的封装。** 说明：前端路线图不替代 iar daemon 或命令行，而是封装调用并提供可视化入口；后端应复用 `agent_runner_orchestrate.py`、`agent_runner_publication.py` 等已有能力，避免重复实现。
9. 合并 PR 这一动作：是由人在前端点击合并，还是 Agent 自动合并，抑或仍然回到 GitHub 上手动合并？（来源：2026-06-14 19:48）
   - **待确认/推荐：先保持人在 GitHub 或前端手动合并，Agent 检测到合并完成后自动高亮下一个。** 理由：当前 iar 流程不自动合并，改动合并权限需要额外安全设计；先不改变现有合并行为，只在前端同步合并完成状态并提示下一个，风险最低。
10. 单个开始是否受全局并行数量限制？当全局队列已满时，单个开始是立即执行还是排队等待？（来源：2026-06-14 19:53）
    - **待确认/推荐：单个开始也受同一并发上限约束，超出时进入等待队列。** 理由：避免用户误点导致 worktree/GitHub API 资源被无限占用；等待队列能提供可预期的执行顺序，也便于统一调度。
11. 依赖关系中「上游已合并到主分支」的判定标准是什么？是 PR merged 事件、分支删除，还是检测主分支已包含上游提交？（来源：2026-06-14 19:53）
    - **待确认/推荐：以 GitHub PR merged 事件/状态为准。** 理由：这是最直接、可观察的里程碑；检测主分支提交容易受 rebase/squash 影响，不够稳定。
